01 / Positioning

项目定位：采集控制面

统一管理目标配置、账号代理、任务调度、运行日志和结果文件，把采集动作纳入可追踪流程。

脚本能力沉淀

保留用户媒体、搜索、评论、主页资料、账号近况等采集能力，避免重写核心逻辑造成风险。

控制面补齐

React + FastAPI 把账号、代理、任务、日志、文件和摘要放到统一入口。

平台化承接复杂度

SQLite 持久化状态，worker 调度任务，runner 子进程隔离执行，结果目录可打包复盘。

02 / Problem Space

平台化问题域：账号、代理、任务与结果

采集链路受账号会话、网络出口、接口变化、目标权限、时间窗口和媒体下载共同影响。

账号会话

Cookie 会过期，X 的校验接口也可能误报。账号状态按可用、待确认、失效分层记录。

网络出口

代理可用性、地理位置、连接稳定性直接影响采集成功率。

接口变化

GraphQL 字段和内部接口可能变化，失败分类用于快速定位。

任务复盘

没有日志、配置快照和结果摘要，就很难解释 0 结果或部分失败。

并发与限流

并发策略结合账号状态、代理质量和接口限流统一控制。

合规边界

Cookie、版权、隐私、平台规则在系统层明确提示和脱敏。

05 / Runtime Flow

任务如何跑起来：从表单到结果归档

任务配置、账号会话、运行日志、进程状态和输出目录共同形成可追踪执行链路。

创建任务目标、条数、内容类型、账号、代理、时间范围。
规范化URL 转用户名，校验推文链接误填、时间范围和账号状态。
入库tasks.status = queued，生成 output_dir / log_path。
调度worker_loop 按创建顺序拉取任务。

归档CSV、媒体、日志、summary_report 写入任务目录。

06 / Account Session Flow

账号授权链路：本地 Chrome helper

线上页面受浏览器安全模型限制，授权窗口和 Cookie 提取由本地 helper 完成。

申请 token后端生成 5 分钟一次性登录 token。
调用本地前端请求 127.0.0.1:18765/start。
Chrome 登录helper 弹出临时 Profile，让用户完成 X 登录。
提取 Cookie只读取 auth_token 和 ct0，不回传明文密码。
回传后端token 校验通过后保存账号。
健康状态检测失败分 active / unknown / check_failed / expired。
Runner 注入任务执行时写入 account_session.json。

安全收益

不保存账号密码，不在日志中展示 Cookie，页面复制信息时自动脱敏。

稳定收益

旧校验接口 404 不再直接误杀账号，降低新号误判。

扩展方向

BitBrowser / Profile 池可以接入同一账号资源模型。

08 / Worker Scheduling

Worker 调度模型：queued 状态驱动

任务进入 queued，后台线程拉取，子进程执行。这个模型适合单机可靠交付，也保留了未来替换为消息队列的空间。

queued等待 worker 拉取，支持删除和取消。
running记录 started_at 和 process_id。
completed退出码 0，写 summary_report。
partial_failed失败但已有部分结果，保留产物。
failed分类失败原因：网络、限流、接口、目标。
retry网络/限流可自动重试，避免一次抖动定生死。

当前收益

实现成本低，状态可查，任务不会因为页面刷新而丢失。

当前瓶颈

单 worker 顺序执行，吞吐上限受单机和账号/代理限制。

演进方向

抽象 TaskQueue 接口，后续替换 Celery/RQ/Redis Stream。

09 / Runner Dispatch Layer

Runner 分发层：保留脚本资产，统一任务入口

runner 是 Web 平台与历史采集脚本之间的适配层。它把配置、账号和输出路径统一，再按 task_type 调用不同能力。

统一执行协议

python web_runner.py \
  --config task_config.json \
  --account account_session.json \
  --output /data/tasks/...

Web 服务只负责生成快照和启动进程，脚本内部变化不会污染控制面。

11 / Failure Taxonomy

失败治理模型：分类、重试、保留产物

失败按账号、网络、限流、目标、接口变化和部分产出分类，分别决定重试、提示或代码修复。

auth_expired

Cookie 缺失、401/403，提示重新登录或更新账号。

network_failed

连接超时、代理异常，可重试，可换出口。

rate_limited

接口限流，进入冷却、退避和并发控制。

target_unavailable

目标不存在、不可见、权限不足，不应盲目重试。

api_changed

字段结构变化，进入代码修复流程。

partial_failed

已有部分产出时保留结果，不让一次异常抹掉有效数据。

12 / Resource Governance

账号池与代理池：采集系统的资源治理模型

账号和代理作为运行资源管理，记录健康状态、失败原因、检测时间和可用性边界。

账号池

  • 状态从 active / unknown / check_failed / expired 分层，避免校验接口 404 误杀账号。
  • 本地 Chrome 登录只保存 auth_token/ct0，日志和复制信息全部脱敏。
  • BitBrowser 导入适合已登录 Profile，后续可扩展为 Profile 池。

代理池

  • 支持 http、socks、host:port:user:pass，统一规范化成 httpx 可用格式。
  • 检测出口 IP，记录 detected_ip、failure_count、last_error。
  • 未来可做质量评分、区域标签、冷却时间和自动熔断。

14 / Performance Roadmap

性能优化路线：背压、冷却与分池

采集系统的性能瓶颈集中在账号、代理、限流、媒体下载和接口稳定性。

当前可优化点

媒体下载并发 max_concurrent_requests

账号冷却与限流退避

代理质量评分

任务分片与批量调度

演进策略

  • 引入 Backpressure：账号/代理失败率高时自动降并发。
  • 分离下载池和接口请求池，避免媒体下载拖慢 GraphQL 请求。
  • 对目标账号做游标缓存和断点续采。
  • 抽象任务队列，为多 worker 和分布式调度做准备。
